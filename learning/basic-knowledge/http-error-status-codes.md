# HTTP エラーステータスコードの基礎

HTTP ステータスコードは、ブラウザや API クライアントからのリクエストに対して、サーバーが処理結果を 3 桁の番号で返す仕組みである。

エラー調査では、まず「何番台か」を見る。細かい原因はシステムごとに違うが、番号台で見ると調査の方向を決めやすい。

## 番号台の見方

| 番号台 | 分類 | 意味 | 調査の方向 |
| --- | --- | --- | --- |
| `1xx` | 情報 | リクエスト処理中の中間応答 | 通常の業務画面では意識することは少ない。 |
| `2xx` | 成功 | リクエストが成功した | エラーではない。想定したデータが返っているか確認する。 |
| `3xx` | リダイレクト | 別 URL への移動やキャッシュ利用 | URL、認証後遷移、リダイレクト設定を確認する。 |
| `4xx` | クライアントエラー | リクエスト側に原因がある可能性が高い | URL、入力値、認証、権限、HTTP メソッド、リクエスト形式を確認する。 |
| `5xx` | サーバーエラー | サーバー側に原因がある可能性が高い | サーバーログ、例外、DB、外部 API、インフラ状態を確認する。 |

## よく見る 4xx エラー

| コード | 名前 | 何のエラーか | 一般的な対処 |
| --- | --- | --- | --- |
| `400` | Bad Request | リクエストの形式が不正。JSON 形式、必須項目、型、桁数などが合っていない。 | request body、query parameter、header、JSON 構文、バリデーションエラー内容を確認する。 |
| `401` | Unauthorized | 認証されていない。ログインしていない、トークンがない、トークンが期限切れ。 | ログイン状態、`Authorization` ヘッダー、アクセストークン、有効期限を確認する。 |
| `403` | Forbidden | 認証はされているが、権限がない。 | ユーザー権限、ロール、アクセス制御、対象データの所有者条件を確認する。 |
| `404` | Not Found | 指定した URL、API エンドポイント、または対象データが見つからない。 | URL の誤字、ルーティング、ID の存在、環境違い、削除済みデータかを確認する。 |
| `405` | Method Not Allowed | URL は存在するが、HTTP メソッドが違う。例: `GET` ではなく `POST` が必要。 | API 仕様書を見て、`GET`、`POST`、`PUT`、`DELETE` などのメソッドを確認する。 |
| `408` | Request Timeout | クライアントからのリクエスト待ちでタイムアウトした。 | ネットワーク状態、送信データ量、クライアント側のタイムアウト設定を確認する。 |
| `409` | Conflict | データの状態が競合している。更新競合、二重登録、楽観ロック失敗など。 | 最新データを再取得し、バージョン番号、更新日時、重複キーを確認する。 |
| `410` | Gone | 以前は存在したリソースが恒久的に削除された。 | 古いリンクや削除済みデータを参照していないか確認する。 |
| `413` | Content Too Large | リクエスト本文が大きすぎる。ファイルアップロードなどで発生しやすい。 | アップロードサイズ、サーバーの上限、プロキシの上限を確認する。 |
| `415` | Unsupported Media Type | `Content-Type` がサーバーの想定と違う。 | `Content-Type: application/json` など、API が要求する形式になっているか確認する。 |
| `422` | Unprocessable Content | 形式は読めるが、業務的・意味的に処理できない。入力値の意味エラーで使われることがある。 | バリデーション結果、業務ルール、必須項目、項目間の組み合わせを確認する。 |
| `429` | Too Many Requests | 短時間にリクエストを送りすぎた。レート制限。 | 連打、リトライ間隔、バッチ処理件数、`Retry-After` ヘッダーを確認する。 |

## よく見る 5xx エラー

| コード | 名前 | 何のエラーか | 一般的な対処 |
| --- | --- | --- | --- |
| `500` | Internal Server Error | サーバー内部で想定外のエラーが発生した。最も汎用的なサーバーエラー。 | アプリケーションログ、例外スタックトレース、直前のリリース、入力値を確認する。 |
| `501` | Not Implemented | サーバーがその機能やメソッドに対応していない。 | API の実装有無、メソッド、ルーティング、リリース状況を確認する。 |
| `502` | Bad Gateway | ゲートウェイやプロキシが、上流サーバーから不正な応答を受け取った。 | ロードバランサー、リバースプロキシ、上流 API、接続先サーバーの状態を確認する。 |
| `503` | Service Unavailable | サービスが一時的に利用できない。メンテナンス、過負荷、起動中など。 | サーバー稼働状況、メンテナンス情報、負荷、再起動、`Retry-After` ヘッダーを確認する。 |
| `504` | Gateway Timeout | ゲートウェイやプロキシが、上流サーバーから時間内に応答を得られなかった。 | 上流 API、DB、外部サービス、タイムアウト設定、遅い処理を確認する。 |

## 調査の進め方

### 1. まず番号台を見る

- `4xx` なら、まずリクエスト側を疑う。
- `5xx` なら、まずサーバー側を疑う。
- `3xx` なら、リダイレクト先や認証後の遷移を確認する。

### 2. リクエスト内容を確認する

API のエラーでは、次を確認する。

| 確認項目 | 見る内容 |
| --- | --- |
| URL | パス、環境、ID、query parameter が正しいか。 |
| HTTP メソッド | `GET`、`POST`、`PUT`、`DELETE` が API 仕様と一致しているか。 |
| Header | `Authorization`、`Content-Type`、独自ヘッダーが正しいか。 |
| Body | JSON 形式、必須項目、型、桁数、日付形式が正しいか。 |
| 認証・権限 | ログイン状態、トークン期限、ロール、アクセス対象が正しいか。 |

### 3. レスポンス本文を見る

同じ `400` や `500` でも、レスポンス本文に詳細なエラーコードやメッセージが入っていることがある。

例:

```json
{
  "code": "VALIDATION_ERROR",
  "message": "name is required"
}
```

この場合、HTTP ステータスコードだけで判断せず、アプリケーション独自の `code` や `message` も確認する。

### 4. サーバーログと突き合わせる

`5xx` の場合は、画面に出たエラーだけでは原因が分からないことが多い。発生時刻、ユーザー ID、リクエスト ID、対象データ ID を手がかりに、サーバーログを確認する。

## 実務での切り分け例

| 状況 | ありそうな原因 | 最初に見るところ |
| --- | --- | --- |
| 画面を開いたら `404` | URL が違う、画面ルートが未定義、対象データが存在しない | URL、ルーティング、データ ID |
| 保存ボタンで `400` | 入力値や JSON 形式が API 仕様と違う | request body、バリデーション結果 |
| ログイン後の API が `401` | トークン未送信、期限切れ | `Authorization` ヘッダー、ログイン状態 |
| 管理者画面で `403` | ユーザーに権限がない | ロール、権限設定 |
| 更新時に `409` | 他ユーザーが先に更新した | version、updated_at、楽観ロック |
| ファイルアップロードで `413` | ファイルサイズが上限を超えている | アップロード上限、プロキシ設定 |
| API が `500` | サーバー内部例外 | アプリケーションログ、例外内容 |
| API が `502` | 上流サービスやプロキシの問題 | gateway、load balancer、上流 API |
| API が `504` | DB や外部 API の応答が遅い | 処理時間、タイムアウト、依存先 |

## 使い分けで迷いやすいコード

| コード | 似ているコード | 違い |
| --- | --- | --- |
| `401` | `403` | `401` は未認証、`403` は認証済みだが権限なし。 |
| `400` | `422` | `400` はリクエスト形式の不正、`422` は形式は読めるが内容・業務ルール上処理できない場合に使われることが多い。 |
| `404` | `410` | `404` は見つからない、`410` は恒久的に削除済み。 |
| `409` | `422` | `409` は現在のサーバー状態との競合、`422` は入力内容の意味的な不正。 |
| `500` | `502` | `500` は処理しているサーバー内部の問題、`502` は中継先から不正な応答を受けた問題。 |
| `503` | `504` | `503` はサービスが一時的に処理できない、`504` は中継先の応答待ちがタイムアウト。 |

## 覚え方

- `400` 番台は「こちらの投げ方が悪いかもしれない」。
- `500` 番台は「サーバーや依存先で問題が起きているかもしれない」。
- `401` はログイン・認証、`403` は権限。
- `404` は場所や対象がない。
- `409` は更新競合。
- `429` は送りすぎ。
- `500` はサーバー内部エラー。
- `502`、`503`、`504` はインフラや外部連携でもよく見る。
