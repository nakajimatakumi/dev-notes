# 要件定義書の書き方

ここで扱う要件定義書は、「何を実現するか」「なぜ必要か」「どの条件を満たせば完成と判断できるか」を、業務とシステムの両面から合意するための文書である。実装方法を細かく決める前に、対象範囲、利用者、業務ルール、必要な機能、品質、制約を明確にする。

目的は、顧客・業務担当・設計者・実装者・テスト担当の間で、作るものと作らないものの認識を合わせること。要件には識別子と受入条件を付け、画面項目定義、イベント定義、API 仕様、テストケースへ追跡できるようにする。

## 何に使うか

| 用途 | 説明 |
| --- | --- |
| 課題と目的の共有 | 現状の問題と、システム化によって達成したい状態を共有する。 |
| 対象範囲の合意 | 今回実現することと、対象外にすることを明確にする。 |
| 機能の合意 | 利用者がシステムで何をできる必要があるかを整理する。 |
| 業務ルールの明確化 | 権限、状態遷移、計算、判定などのルールを一意にする。 |
| 品質目標の合意 | 性能、可用性、セキュリティなどの測定可能な目標を決める。 |
| 見積もりと計画 | 対象機能、外部連携、データ移行、制約から作業量を見積もる。 |
| 設計・テストの起点 | 各要件を画面、イベント、API、テストケースへつなげる。 |
| 変更管理 | 要件変更の理由と影響範囲を追跡できるようにする。 |

## 要件定義と設計の違い

要件定義では、利用者や業務に必要な結果と制約を定める。設計では、その要件をどのような画面、API、データ構造、処理方式で実現するかを定める。

| 要件定義で決めること | 設計で決めること |
| --- | --- |
| 利用者が申請を検索できること | 検索画面の項目配置や検索 API の仕様 |
| 申請者だけが下書きを編集できること | 権限判定を行う処理や実装箇所 |
| 検索結果を3秒以内に表示すること | インデックス、キャッシュ、ページング方式 |
| 承認済みデータは変更できないこと | ボタン制御、API 側の検証、エラーコード |

実現方法に強い制約がある場合は要件として記載してよいが、その制約が必要な理由も併記する。

## 要件の粒度

1つの要件は、1つの能力、業務ルール、または品質条件として扱う。要件 ID を付け、単独で必要性と達成可否を判断できる粒度にする。

例:

- `FR-001`: 申請者は、自分が作成した下書き申請を編集できること。
- `FR-002`: 承認済みの申請は、すべての利用者が編集できないこと。
- `NFR-001`: 通常負荷時、検索結果を操作開始から3秒以内に表示すること。

「使いやすいこと」「適切に制御すること」のように、達成可否を判定できない表現は避ける。誰が、どの条件で、何をできるか、またはどの数値を満たすかを具体的に書く。

## 必要な項目

| 項目       | 書く内容                  | 必要な理由                  |
| -------- | --------------------- | ---------------------- |
| 背景・課題    | 現行業務の問題、依頼の経緯         | なぜ開発するのかを判断できるようにするため。 |
| 目的・成功指標  | 実現したい状態と測定方法          | 完成後に効果を評価するため。         |
| ステークホルダー | 利用者、業務責任者、運用者、連携先     | 誰の要求と合意が必要かを明確にするため。   |
| 対象範囲     | 対象業務、機能、組織、データ        | 見積もりと責任範囲を明確にするため。     |
| 対象外      | 今回は実現しない事項            | 暗黙の期待や認識違いを防ぐため。       |
| 前提・制約    | 期限、予算、既存環境、利用技術、法令    | 実現方法や計画に影響する条件を共有するため。 |
| 業務フロー    | 現状と変更後の業務手順           | システムが支援する範囲を明確にするため。   |
| 機能要件     | 利用者や他システムが必要とする機能     | 実装対象を具体化するため。          |
| 業務ルール    | 権限、状態、計算、判定、期限        | 担当者ごとの解釈差をなくすため。       |
| データ要件    | 管理対象、必須項目、保持期間、件数     | 画面、API、データ設計の前提にするため。  |
| 外部連携要件   | 連携先、タイミング、データ、失敗時の扱い  | システム境界と依存関係を明確にするため。   |
| 非機能要件    | 性能、可用性、セキュリティ、運用など    | 機能以外の品質を検証可能にするため。     |
| 移行・運用要件  | 初期データ、監視、バックアップ、問い合わせ | 稼働開始と稼働後に必要な作業を定めるため。  |
| 受入条件     | 要件を満たしたと判定する具体的な条件    | テストと受入判定の基準にするため。      |
| トレーサビリティ | 要件と設計・テスト成果物の対応       | 実装漏れと不要実装を確認するため。      |

## 記入例: 申請管理機能

### 背景・課題

- 申請内容を表計算ファイルとメールで管理しており、最新版の判別に時間がかかる。
- 承認状況を申請者が確認できず、担当者への問い合わせが発生している。
- 承認済みデータを誤って更新するリスクがある。

### 目的・成功指標

| 目的 | 成功指標 | 測定方法 |
| --- | --- | --- |
| 申請と承認状況を一元管理する | 対象申請の100%をシステムに記録する | 月次の申請台帳とシステム件数を照合する |
| 状況確認の問い合わせを減らす | 稼働3か月後に問い合わせ件数を稼働前比50%以下にする | 問い合わせ管理表を月次集計する |

### 対象範囲

#### 対象

- 申請の新規作成、下書き保存、提出、承認、差し戻し、状態確認。
- 申請者、承認者、業務管理者による利用。
- 稼働開始日以降に登録された申請データ。

#### 対象外

- 承認後に行う支払処理。
- 稼働開始前の申請データの移行。
- 組織改編に伴う承認経路の自動変更。

### 機能要件

| 要件 ID | 要件 | 利用者・契機 | 優先度 | 受入条件 |
| --- | --- | --- | --- | --- |
| FR-001 | 申請者は、新しい申請を作成し、下書きとして保存できること。 | 申請者が新規作成を選択したとき | 必須 | 必須項目を入力して保存すると、状態が「下書き」の申請が1件作成され、再表示できる。 |
| FR-002 | 申請者は、自分が作成した下書き申請を編集できること。 | 申請者が下書きを選択したとき | 必須 | 作成者本人かつ状態が「下書き」の場合だけ変更を保存できる。 |
| FR-003 | 承認者は、提出された申請を承認または差し戻しできること。 | 承認者が処理対象を選択したとき | 必須 | 承認すると状態が「承認済み」、差し戻すと状態が「差し戻し」になり、処理日時と処理者が記録される。 |
| FR-004 | 利用者は、権限の範囲内で申請の現在状態と処理履歴を確認できること。 | 利用者が申請詳細を表示したとき | 重要 | 現在状態と、各処理の日時・処理者・処理種別が時系列で表示される。 |

### 業務ルール

| ルール ID | 条件 | ルール | 例外 |
| --- | --- | --- | --- |
| BR-001 | 申請の状態が「下書き」 | 作成者本人だけが編集できる。 | 業務管理者も内容を変更できない。 |
| BR-002 | 申請の状態が「提出済み」 | 指定された承認者だけが承認または差し戻しできる。 | なし。 |
| BR-003 | 申請の状態が「承認済み」 | すべての利用者が申請内容を変更できない。 | 取消機能は今回の対象外。 |

### 非機能要件

| 要件 ID | 分類 | 要件 | 測定条件・受入基準 |
| --- | --- | --- | --- |
| NFR-001 | 性能 | 申請検索の結果を3秒以内に表示すること。 | 同時利用50人、検索対象10万件、検索結果100件以下の条件で、操作開始から結果描画完了までの95パーセンタイルが3秒以内。 |
| NFR-002 | セキュリティ | 利用者は自分に許可された申請だけを参照できること。 | 申請者、承認者、業務管理者の権限別テストで、許可されていない申請を画面とAPIのどちらからも取得できない。 |
| NFR-003 | 監査 | 申請の作成、提出、承認、差し戻しを記録すること。 | 操作種別、対象申請、処理者、処理日時、処理結果を検索可能な監査ログとして1年間保持する。 |

## 画面項目定義・イベント定義とのつなぎ方

要件定義は「必要な能力と条件」、画面項目定義は「表示・入力する情報」、イベント定義は「操作時の処理」を担当する。

| 要件定義 | 画面項目定義 | イベント定義 |
| --- | --- | --- |
| `FR-001` 下書きを保存できる | 申請名、申請内容、保存ボタンの表示・入力規則 | 保存ボタン押下時のチェック、API 順序、成功・失敗時処理 |
| `BR-003` 承認済みは変更不可 | 承認済みの場合の入力可否 | 初期表示時と保存時の編集可否判定 |
| `NFR-001` 検索結果を3秒以内に表示 | 検索条件と結果件数の表示 | 検索イベントの開始・完了条件、タイムアウト時処理 |

各詳細文書に要件 ID を記載し、要件から実装とテストまでたどれる状態にする。

## レビュー観点

- 背景、課題、目的のつながりが説明できるか。
- 対象と対象外が、業務・機能・利用者・データの観点で明確か。
- 各機能要件に利用者、条件、必要な動作が書かれているか。
- 「適切に」「必要に応じて」など、判定できない表現が残っていないか。
- 業務ルールの条件、許可・禁止、例外が一意に読めるか。
- 非機能要件に数値、測定条件、判定基準があるか。
- 正常系だけでなく、権限不足、入力不備、競合、連携失敗を扱っているか。
- データの発生元、保持期間、機密区分、削除条件が決まっているか。
- 各要件に受入条件があり、テストで達成可否を判断できるか。
- 要件 ID から画面、イベント、API、テストケースを追跡できるか。
- 前提が崩れた場合や要件が変更された場合の影響範囲を確認できるか。
- 顧客と開発チームが、同じ完成状態を想像できるか。
