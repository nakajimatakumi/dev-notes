# 画面イベント定義書の書き方

ここで扱うイベント定義書は、画面や業務操作を起点にした処理定義である。たとえば「XXボタン押下時」「画面初期表示時」「プルダウン変更時」に、どの API をどの順番で呼び、どの条件で分岐し、画面をどう更新するかを明確にする。

目的は、実装者・設計者・レビュー担当・顧客との間で、処理順序と業務ルールの認識を合わせること。特に、1つの操作で複数 API を同期的に呼ぶ画面では、イベント単位の定義書があると実装漏れや見積もり漏れを減らせる。

## 何に使うか

| 用途 | 説明 |
| --- | --- |
| 実装時の共通認識 | フロントエンド、バックエンド、レビュー担当が同じ処理順序を見て実装できる。 |
| API 実行順序の明確化 | 判定 API、取得 API、保存 API などの呼び出し順と依存関係を整理できる。 |
| 分岐条件の整理 | API 結果や画面状態によって、次に何をするかを明確にできる。 |
| 工数見積もり | 1イベント内の API 数、分岐数、エラー処理、画面更新範囲から作業量を見積もれる。 |
| 顧客説明・合意 | ボタン押下時に何が起きるかを業務目線で説明できる。 |
| テスト観点の抽出 | 正常系、異常系、分岐、二重押下、API 失敗時の確認項目を作りやすい。 |

## イベントの粒度

画面イベント定義では、「ユーザー操作」または「画面ライフサイクル」を1イベントとして扱う。

例:

- `検索ボタン押下時`
- `保存ボタン押下時`
- `画面初期表示時`
- `明細行選択時`
- `ステータス変更時`

1イベントの中に複数 API が入ってよい。むしろ、複数 API をどの順で呼ぶか、途中で止める条件は何かを書くための定義書と考える。

## 必要な項目

| 項目 | 書く内容 | 必要な理由 |
| --- | --- | --- |
| イベント名 | `保存ボタン押下時` など | どの画面操作の話かを明確にするため。 |
| 対象画面 | 画面名、画面 ID、対象機能 | 同じボタン名でも画面によって処理が違うため。 |
| 発生契機 | ユーザー操作や画面表示など | 処理がいつ開始されるかを明確にするため。 |
| 事前条件 | 入力値、選択状態、権限、画面状態 | API を呼ぶ前に満たすべき条件を共有するため。 |
| 入力チェック | 必須、形式、桁数、業務チェック | API 呼び出し前に止める条件を明確にするため。 |
| API 実行順序 | 呼び出す API と順番 | 実装順序、依存関係、見積もりを明確にするため。 |
| 分岐条件 | API 結果や状態による if/else | どの条件で次の API に進むか、処理を止めるかを決めるため。 |
| API 入出力 | request に渡す値、response から使う値 | 画面項目と API 項目の対応を明確にするため。 |
| 成功時処理 | メッセージ、画面遷移、再検索、状態更新 | ユーザーに何が起きたように見えるかを定義するため。 |
| 失敗時処理 | エラー表示、リトライ、処理中断 | 障害時の挙動を実装者ごとに変えないため。 |
| 排他・二重実行 | ボタン disable、連打防止、楽観ロック | 二重登録や競合更新を防ぐため。 |
| ログ・監査 | 操作ログ、業務ログ、エラーログ | 問い合わせや障害調査に使うため。 |
| テスト観点 | 正常系、異常系、境界値、分岐 | 定義からテストケースを作れるようにするため。 |

### `<イベント名>`

#### 基本情報

| 項目 | 内容 |
| --- | --- |
| 対象画面 | `<画面名>` |
| 発生契機 | `<ボタン押下 / 初期表示 / 選択変更>` |
| 利用者 | `<利用者ロール>` |
| 目的 | `<この操作で実現したい業務目的>` |

#### 事前条件

- `<対象データが選択されていること>`
- `<必須項目が入力済みであること>`
- `<ユーザーに実行権限があること>`

#### 入力チェック

| No | チェック内容 | NG 時の処理 |
| --- | --- | --- |
| 1 | `<必須チェック>` | `<メッセージを表示して処理終了>` |
| 2 | `<業務チェック>` | `<メッセージを表示して処理終了>` |

#### API 実行フロー

| 順序 | API | 目的 | 実行条件 | 成功時 | 失敗時 |
| --- | --- | --- | --- | --- | --- |
| 1 | `<判定API>` | `<実行可否を判定>` | `<入力チェック OK>` | `<次の API へ>` | `<エラー表示して終了>` |
| 2 | `<状態取得API>` | `<現在状態を取得>` | `<判定 OK>` | `<保存 API へ>` | `<エラー表示して終了>` |
| 3 | `<保存API>` | `<変更内容を保存>` | `<保存可能状態>` | `<完了処理へ>` | `<エラー表示して終了>` |

#### 分岐ルール

| 条件 | 処理 |
| --- | --- |
| `<判定API.result = NG>` | `<メッセージを表示して以降の API は呼ばない>` |
| `<状態取得API.status = locked>` | `<排他エラーを表示して保存しない>` |
| `<保存API 成功>` | `<完了メッセージを表示し、一覧を再取得>` |

#### API 入出力

| API | request に渡す値 | response から使う値 |
| --- | --- | --- |
| `<判定API>` | `<画面項目A, ユーザーID>` | `<判定結果, メッセージ>` |
| `<状態取得API>` | `<対象ID>` | `<現在ステータス, 更新日時>` |
| `<保存API>` | `<対象ID, 入力値, 更新日時>` | `<保存結果, 新しい更新日時>` |

#### 成功時処理

- `<完了メッセージを表示する>`
- `<画面を参照モードに戻す>`
- `<一覧または詳細を再取得する>`

#### 失敗時処理

| 失敗箇所 | 表示・処理 |
| --- | --- |
| 入力チェック | `<項目下にエラー表示し、API は呼ばない>` |
| 判定 API | `<業務エラーメッセージを表示し、処理終了>` |
| 状態取得 API | `<システムエラーを表示し、処理終了>` |
| 保存 API | `<保存失敗メッセージを表示し、入力内容は保持>` |

#### 排他・二重実行

- 処理中は `<実行ボタン>` を disable にする。
- 保存 API には `<更新日時 / version>` を渡し、楽観ロックを行う。
- 二重押下時に同じ保存が複数回走らないようにする。

#### テスト観点

- 正常に全 API が順番通り呼ばれること。
- 判定 API が NG の場合、後続 API を呼ばないこと。
- 状態取得 API で保存不可状態の場合、保存 API を呼ばないこと。
- 保存 API 失敗時、入力内容が消えないこと。
- 処理中にボタンを連打しても二重保存されないこと。

## 記入例: 保存ボタン押下時

#### 基本情報

| 項目 | 内容 |
| --- | --- |
| 対象画面 | 申請詳細画面 |
| 発生契機 | 保存ボタン押下 |
| 利用者 | 申請者 |
| 目的 | 申請内容を保存し、最新状態を画面に反映する |

#### 事前条件

- 申請データが画面に表示されていること。
- 必須項目が入力済みであること。
- ログインユーザーが対象申請を編集できること。

#### API 実行フロー

| 順序 | API | 目的 | 実行条件 | 成功時 | 失敗時 |
| --- | --- | --- | --- | --- | --- |
| 1 | 編集可否判定 API | ユーザーが保存可能か判定する | 入力チェック OK | 現在状態取得 API へ | メッセージ表示して終了 |
| 2 | 現在状態取得 API | 申請の最新ステータスを取得する | 編集可 | 保存 API へ | エラー表示して終了 |
| 3 | 申請保存 API | 入力内容を保存する | ステータスが `下書き` | 完了処理へ | エラー表示して終了 |

#### 分岐ルール

| 条件 | 処理 |
| --- | --- |
| 編集可否判定 API が `不可` | 理由メッセージを表示し、現在状態取得 API 以降は呼ばない。 |
| 現在ステータスが `承認済み` | 「承認済みのため保存できません」を表示し、保存 API は呼ばない。 |
| 保存 API が成功 | 「保存しました」を表示し、申請詳細を再取得する。 |

#### API 入出力

| API | request に渡す値 | response から使う値 |
| --- | --- | --- |
| 編集可否判定 API | `application_id`, `user_id` | `editable`, `reason_message` |
| 現在状態取得 API | `application_id` | `status`, `version`, `updated_at` |
| 申請保存 API | `application_id`, 入力項目, `version` | `saved`, `new_version` |

#### 成功時処理

- 完了メッセージ「保存しました」を表示する。
- 保存後の `new_version` を画面状態に反映する。
- 必要に応じて詳細情報を再取得する。

#### 失敗時処理

| 失敗箇所 | 表示・処理 |
| --- | --- |
| 入力チェック | エラー項目を強調表示し、API は呼ばない。 |
| 編集可否判定 API | API が返した理由メッセージを表示する。 |
| 現在状態取得 API | 「最新状態を取得できませんでした」を表示する。 |
| 申請保存 API | 「保存に失敗しました」を表示し、入力内容は保持する。 |

#### 排他・二重実行

- 保存処理中は保存ボタンを disable にする。
- 申請保存 API には `version` を渡して楽観ロックする。
- version 不一致の場合は、再読み込みを促すメッセージを表示する。

## レビュー観点

- イベント名から対象操作が分かるか。
- API の実行順序が一意に読めるか。
- 途中で処理を止める条件が書かれているか。
- 各 API の request に必要な画面項目が分かるか。
- response のどの値を次の判定に使うか分かるか。
- 正常系だけでなく、API ごとの失敗時処理があるか。
- 二重押下、排他、楽観ロックの扱いが書かれているか。
- 顧客が読んでも業務上の流れを理解できるか。
