# リポジトリ運用ガイド

## プロジェクト構成

このリポジトリは Markdown ベースの Obsidian ノート保管庫です。長期的に残すノートは `learning/` 配下に置き、トピックごとに分類します。現在は `learning/architecture/` と `learning/ai/` があります。

新しいノートは、近い既存トピックの配下に追加してください。該当する分類がない場合は、`learning/<topic>/` のように小文字・ハイフン区切りのディレクトリを作成します。

`.obsidian/` は Obsidian のワークスペースや vault 設定用です。意図した設定変更でない限り、手動編集や不要な差分は避けてください。

## 開発・確認コマンド

アプリケーションではないため、ビルドや実行コマンドはありません。作業前後の確認には次を使います。

- `rg --files`: リポジトリ内のファイルを素早く一覧する。
- `git status --short`: 変更・未追跡ファイルを確認する。
- `git diff -- <path>`: 特定ファイルの差分を確認する。

Markdown の見た目が重要な場合は、Obsidian または Markdown プレビューで確認してください。

## 文書スタイルと命名規則

Markdown は見出しを明確にし、短い段落で書きます。ページタイトルは `#`、主要セクションは `##` を使います。参照しやすい内容は表にまとめても構いません。

ファイル名とディレクトリ名は小文字・ハイフン区切りを基本にします。例: `learning/ai/codex-cli-slash-commands.md`

日本語と英語の本文は UTF-8 で保存します。既存ノートの言語や粒度に合わせ、関連しない整形変更は避けてください。

## テスト・レビュー方針

自動テストはありません。コミット前に、変更した Markdown の見出し、表、リンク、コードフェンスが正しく表示されることを確認します。

コマンド例を書く場合は、このリポジトリと現在の環境で意味が通るものにしてください。外部サービスや最新仕様に依存する内容は、出典や取得日を残すと後から見直しやすくなります。

## コミットと Pull Request

最近の履歴には `vault backup: 2026-07-05 23:23:47` のような短い説明型コミットがあります。通常の変更では、`add codex cli slash command notes` のように内容が分かる短いメッセージを使います。

Pull Request では、追加・変更したノート、移動したパス、`.obsidian/` への変更有無を明記してください。スクリーンショットは、Obsidian 上の表示や Markdown レンダリングがレビュー対象になる場合だけ添付します。

## エージェント向け指示

ルートのガイドファイルを作成・編集する前に、既存ファイルの有無を確認してください。既存の contributor 指示は、明示的な依頼なしに上書きしないでください。

生成するノートは目的に絞り、事実が変わりうる内容では出典や確認日を残します。依頼範囲外の整理、リネーム、メタデータ変更は避けてください。

## 質問ログ

- このプロジェクト内でユーザーが質問したときは、Asia/Tokyo の日付に基づき `question-logs/YYYY-MM-DD.md` の末尾へ追記してください。
- 日付ファイルがなければ作成し、過去の記録は上書き・書き換えしないでください。
- ローカル時刻、ユーザーの質問本文、回答の短い要約を、ユーザーが使用した言語で記録してください。
- 依頼と質問が同じメッセージに含まれる場合は、メッセージ全文を記録してください。
- ツール出力、非公開の指示、認証情報、アクセストークンなどの秘密情報は記録しないでください。質問に秘密情報が含まれる場合は、その値だけを `[REDACTED]` に置き換えてください。
- 実作業の完了後に回答要約を確定し、未完了の場合は阻害要因または未解決点を記録してください。
